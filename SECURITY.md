# セキュリティ・プライバシー設計

## 1. 適用範囲と現在の状態

このリポジトリは、くらしDXサポートの静的公開版です。診断はブラウザ内だけで計算し、相談予約は電話アプリへ引き継ぎます。氏名・電話番号・相談内容は画面で取得しません。Cookie、Local Storage、Session Storage、IndexedDBも使用しません。

オンライン予約、相談履歴、管理画面を追加する場合は、以下の「機能追加ゲート」を満たすバックエンド・認証・監視基盤が必要です。現在のデータ非取得方針を、承認なく変更してはいけません。

## 2. データ分類

| クラス | 例 | 取得 | 保存 | AI入力 |
|---|---|---:|---:|---:|
| 公開 | サービス説明、料金例 | 可 | 可 | 承認済みサービスのみ可 |
| 内部 | 匿名化済み集計、運用手順 | 必要最小限 | 規程に従う | 承認・目的確認後のみ |
| 個人情報 | 電話番号、氏名、相談履歴 | 明示同意時のみ | 暗号化・30日以内を標準 | 原則禁止 |
| 高機密 | パスワード、暗証番号、本人確認書類、カード情報 | 取得禁止 | 保存禁止 | 入力禁止 |

データ項目を追加する場合、プロダクト責任者とセキュリティ責任者が、利用目的、法的根拠、保存期間、閲覧者、委託先、削除方法を承認します。

## 3. 公開版で実装済みの防御

- 外部通信を `connect-src 'none'` で禁止
- スクリプト、CSS、画像を同一オリジンに限定
- 外部Webフォントと外部JavaScriptを不使用
- フォーム送信を `form-action 'none'` で禁止
- フレーム埋め込みを禁止し、クリックジャッキングを抑止
- カメラ、マイク、位置情報、決済、USBの権限を無効化
- `Referrer-Policy: no-referrer` と `Cache-Control: no-store`
- 診断結果を永続保存せず、旧版が残した既知のLocal Storageキーを削除
- 動的な数値表示は `textContent` / `replaceChildren` を使い、HTML文字列を挿入しない
- 相談予約では個人情報を取得せず、電話アプリへ引き継ぐ
- 利用者向け画面と運営者向けのセキュリティ文書を分離

## 4. 脅威モデル

| 脅威 | 主な影響 | 必須対策 |
|---|---|---|
| 個人情報の過剰取得 | 漏えい時の被害拡大 | データ最小化、入力禁止項目、保存期限 |
| XSS・入力値注入 | 情報窃取、画面改ざん | CSP、出力エンコード、サーバー側検証、テンプレート自動エスケープ |
| 不正な予約送信 | 業務妨害、なりすまし | CSRF対策、レート制限、Bot対策、再認証 |
| アカウント侵害 | 相談履歴・連絡先の閲覧 | SSO/MFA、最小権限、条件付きアクセス、セッション失効 |
| 監査ログへの個人情報混入 | 二次漏えい | PIIマスキング、本文非記録、アクセス制御、保持期限 |
| 委託先・AIへの流出 | 第三者提供、学習利用 | 承認済みサービス、DPA、学習不使用、保存地域確認 |
| 依存パッケージ汚染 | 認証情報窃取、改ざん | 依存最小化、固定バージョン、SBOM、署名・ハッシュ検証 |
| 権限情報の剥落 | 本来見えない相談記録の露出 | データソースのACL継承、認可テスト、検索結果の権限フィルタ |

## 5. オンライン予約・管理機能の追加ゲート

### 通信・保存

- TLS 1.2以上（推奨1.3）、HSTS、管理された証明書更新
- 個人情報をKMS管理鍵で保存時暗号化し、鍵をアプリから分離
- 予約データは目的達成後30日以内に自動削除。延長は理由と承認を記録
- バックアップにも同じ削除期限・暗号化・アクセス制御を適用
- クライアントコード、ログ、リポジトリに秘密情報を置かない

### 認証・認可・監査

- 管理者・相談員はSSOとMFAを必須化
- 役割別権限、最小権限、四半期ごとの権限棚卸し
- 管理操作・閲覧・出力・削除を監査ログへ記録
- 監査ログは改ざん防止保管し、電話番号や相談本文を記録しない
- 退職・異動・委託終了時の即時失効を自動化

### アプリケーション

- サーバー側の入力検証、CSRF対策、レート制限、サイズ上限
- SAST、DAST、秘密情報スキャン、依存関係スキャンをCIで必須化
- SBOMをリリースごとに生成し、High/Critical脆弱性があれば公開を停止
- 管理画面と公開画面を分離し、管理画面はネットワーク制限を追加
- エラーメッセージやログへ入力値、スタックトレース、秘密情報を出さない

### AI・委託先ガバナンス

- AI/SaaSは申請、セキュリティレビュー、オーナー登録後のみ利用
- 個人情報をモデル学習・製品改善に使わない契約を基本条件とする
- AIへ入力できるデータ分類を明文化し、高機密・個人情報は原則禁止
- プロンプトインジェクション、誤情報、権利侵害、偏りをリスク評価へ追加
- RAG導入時は元データのアクセス権が検索結果まで継承されることを自動テスト

## 6. インシデント対応

1. 検知・受付: 疑わしい挙動、誤送信、不正閲覧を一元窓口で受け付ける
2. 封じ込め: 対象アカウント・API・データ出力を停止し、証拠を保全する
3. 影響評価: データ種別、件数、対象者、期間、原因、第三者提供を特定する
4. 通知: 法令、契約、社内基準に従い、本人・監督機関・取引先へ期限内に通知する
5. 復旧: 安全性を確認した変更のみ再公開し、監視を強化する
6. 再発防止: 根本原因、是正責任者、期限、検証結果を経営層へ報告する

オンライン予約や管理機能の開始前に、24時間連絡先、指揮責任者、法務・広報・DPO・委託先の連絡網、判断基準、演習計画を確定します。年1回以上の机上演習を実施します。

## 7. 参考資料との対応

JNSA「CISO視点から見るAIの安全な利用とセキュリティ」から、以下を設計へ反映しています。

- 利用サービスの選定基準と承認フロー
- AIで扱うデータの分類
- AIエージェント利用ルール
- SSO/MFA/監査ログ
- RAGのアクセス権継承
- AIを含むリスク評価とインシデント対応
